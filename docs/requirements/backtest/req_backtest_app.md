# 需求文档 (Requirements Document)

## 项目概述 (Project Overview)

在现有 TradingAssistance 基础上进行破坏式重构，建立可扩展的策略回测与参数优化体系；定投与真实投资记录以其他应用接入系统，本项目仅预留对接入口。同时引入共享核心库（数据类型与指标接口）供后续定时服务与其他服务复用，支持可替换 data provider、YAML 分层配置与可视化输出，并优先采用成熟第三方库以减少重复实现。

## 关键评估 (Critical Assessment)

- 架构风险：配置驱动 + 多策略 + 多数据源会提高耦合与调试复杂度，需要严格的接口隔离与校验层。
- 项目膨胀：回测引擎、可视化与配置系统同时推进，范围较大。建议按阶段推进（核心回测与数据层 → 配置与多组合 → 可视化/报告）。

## 范围定义 (Scope)

### 包含 (In Scope)

- 策略回测流程与结果输出，策略/仓位/指标接口可扩展
- YAML 分层配置（多组合 profiles、meta 组合、超参数统一管理）
- 可替换 data provider 与本地缓存/存储策略
- 日线数据 CSV 存储
- 可视化输出（收益曲线、回撤曲线、对比）
- 预留对接定投与真实投资记录服务的接口边界
- 允许同一目录下存在多个项目模块，并通过 CLI args 选择启动模式/入口
- 引入共享核心库以复用数据类型与指标接口

### 排除 (Out of Scope)

- 实盘交易下单或券商对接
- 实时行情推送与复杂告警渠道（短信/邮件/IM）
- 向后兼容旧模块或旧接口
- 定投功能与真实投资记录（迁移到独立项目）

## 非功能性需求 (Non-Functional Requirements)

| 类别 | 要求 |
| :--- | :--- |
| 性能 | 单次回测在常见标的与日频数据下可在可接受时间内完成（如 < 30s，具体待确认） |
| 安全 | 本地数据与日志不应泄露敏感信息（API Key 等） |
| 兼容性 | 不要求向后兼容，允许破坏式重构 |
| 可扩展性 | 策略、频率、通知、存储与 data provider 通过接口扩展，避免侵入核心流程 |
| 可追溯性 | 参数优化结果需单独存储且不覆盖原始配置，默认写入 `outputs/` |
| 共享规范 | shared_core/core_utils 仅允许无状态、低依赖、纯函数工具；禁止引入 app 级业务逻辑 |
| 配置与机密 | 通过 configs 选择 simulator 数据源（本地 CSV）；API key 与敏感信息通过根目录 `.env` 提供并由 `.gitignore` 排除 |
